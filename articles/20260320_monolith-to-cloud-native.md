# モノリスからクラウドネイティブへ — 会話で学ぶアーキテクチャ転換

Django でWebアプリを4本作った開発者が、Google Cloud上にクラウドネイティブなシステムを構築しようとしている。AI（Claude Code）との会話を通じて、モノリスとクラウドネイティブの違いを一つずつ紐解いていく。

---

## この会話の前提

**人間側のプロフィール**:
- Pythonでの開発経験あり（Django Webアプリ4本、デスクトップアプリ1本を設計〜テストまで完了）
- MVCモデル、V字モデルのテスト工程を体系的に実践済み
- 開発環境: Nginx + Gunicorn + Django + PostgreSQL の構成を経験
- Google Cloud資格2つ、Kubernetes資格（CKA/CKAD）を保有
- クラウドは**Google Cloud**を使うことを決めている
- **ただし、クラウドの実務経験はゼロ**。座学で概念を理解している段階
- AI（Claude Code）と一緒に開発する環境を日常的に使用

つまり、**モノリスの実践経験＋クラウドの座学知識**を持つ人間が、実際にクラウドネイティブな構成を構築する前に、AIと対話しながら全体像を整理している会話。

---

## 1. クラウドは「リフト」するだけじゃない

**人間**: Google Cloud上にシステムを構築したい。クラウドとなると、単純にWebシステムを作ってリフトすればいいって話じゃないよね？ネットワーク、セキュリティ、リージョンのミラーリング、Kubernetes、Docker、マイクロサービス……全部やっていかないといけない。

**AI**: そうですね。ざっくり分けると——

- **ネットワーク**: VPC、サブネット、ファイアウォール、ロードバランサー
- **セキュリティ**: IAM、サービスアカウント、Secret Manager
- **可用性**: マルチゾーン/マルチリージョン
- **コンテナ**: Docker、GKE（Kubernetes）
- **アーキテクチャ**: マイクロサービス

**人間**: そういう全体感はわかるんだよ。全体感がわかっていて、あなた（AI）がいるから、できるんじゃないかなって思った。

---

## 2. 「できない」を疑え

**AI**: ただ、GCPのコンソール操作やアカウント設定、課金周り、動作確認は私が直接触れないので、そこは人の手が必要です。

**人間**: GCPの操作は`gcloud` CLIでできるんじゃないの？

**AI**: ……たしかに。

**人間**: VSCodeの拡張機能でコンソールに接続できるよね？

**AI**: Cloud Codeがありますね。

**人間**: ブラウザの動作確認は、あなたPlaywrightでテストツール作れるでしょ？

**AI**: はい、作れます。

**人間**: じゃあ「できない」のは何？

**AI**: ……GCPアカウントの初回作成とクレジットカード登録だけです。

**人間**: そこは人間が行うべきところだからOK。ということは全部お願いできるね。

> **教訓**: AIは「できない」をデフォルトにしがち。「どうすればできるか」を先に考えさせることで、可能性が大きく広がる。

---

## 3. フレームワーク選定 — DjangoからFastAPIへ

**人間**: 開発言語はPython。フレームワークは、コンテナがあるものがいいよね。

**AI**: Pythonの公式Dockerイメージがあるのでどのフレームワークでもコンテナ化できますが、マイクロサービスを視野に入れるなら選択が変わります。

| | Django | FastAPI |
|---|---|---|
| タイプ | フルスタック（モノリス寄り） | API特化（マイクロサービス向き） |
| 処理方式 | 同期 | 非同期（async/await） |
| コンテナサイズ | 大きめ | 軽量 |
| APIドキュメント | 自分で作る | 自動生成（Swagger UI） |

**人間**: クラウドだからDjangoじゃないほうがよさそうだね。FastAPIとFlaskは何が違うの？

**AI**: Flaskは「自由に組み立てるシンプルなフレームワーク」、FastAPIは「モダンなAPI開発に必要なものが最初から揃っているフレームワーク」です。マイクロサービス＋コンテナ＋クラウドならFastAPIが素直にハマります。

---

## 4. MVCモデルとの決別

**人間**: FastAPIのフレームワークって、MVCモデルとは違うの？

**AI**: はい、ちょっと違います。

**Djangoの場合（MVC/MTV）**:
```
ユーザー → URL → View → Model(DB) → Template(HTML) → ユーザー
```

**FastAPIの場合**:
```
ユーザー → エンドポイント → ロジック → Model(DB) → JSONレスポンス → ユーザー
```

FastAPIは**バックエンドAPI専用**で、テンプレート（HTML）を持ちません。画面（フロントエンド）は別で作る前提です。

**人間**: じゃあフロントエンドはReactで作って、FastAPIのところはPythonで作る感じ？

**AI**: そうです。

```
ユーザー → React(画面) ←JSON→ FastAPI(API) ←→ DB
```

---

## 5. 画面遷移 — サーバーからブラウザへ

**人間**: 画面をどんどん遷移させていくとき、今までの感覚だとDjangoでコントロールするじゃん？その役割はどこになるの？

**AI**: ここが一番大きな違いです。

**Django**: ボタンを押す → Djangoに飛ぶ → Djangoが次の画面を作って返す

**React + FastAPI**: ボタンを押す → React自身が画面を切り替える → 必要ならFastAPIにデータだけ取りに行く

画面遷移の制御は**React側（ブラウザ内）**に移ります。FastAPIはデータの出し入れだけに専念します。

**人間**: そういう作りだとメンテナンス難しそうだし、ちゃんとドキュメントに残しておかないと、React内がぐちゃぐちゃになりそう。

**AI**: その感覚は正しいです。Djangoなら`urls.py`を見れば全遷移が一覧できますが、Reactだとコンポーネントの中に埋もれやすい。画面設計書・画面遷移図は、Reactのほうがむしろ重要になります。

---

## 6. 状態管理 — ステートレスの現実

**人間**: クラウドネイティブとしてはステートレスが基本なんだろうけど、画面遷移するから状態保持が必要なところもあるよね。

**AI**: 完全なステートレスは現実的には無理です。ポイントは**状態をどこに持つか**。

| 状態の種類 | 置き場所 |
|-----------|---------|
| 今どの画面か | ReactのRouter（ブラウザのURL） |
| フォーム入力中のデータ | Reactのコンポーネント内（メモリ） |
| ログイン状態 | トークン（ブラウザのストレージ） |
| ユーザーのデータ | DB（Cloud SQL） |

考え方は「**コンテナ自体はステートレス、状態はコンテナの外に出す**」。こうすればコンテナがいつ消えても再起動しても大丈夫で、スケールアウトが自由になります。

---

## 7. 3層構造の変化

**人間**: MVCモデルでいうWebサーバ、APサーバ、DBサーバってクラウドだとどうなるの？

| 従来（Django） | クラウドネイティブ |
|---------------|------------------|
| Webサーバ（Nginx） | Reactコンテナ＋GCPロードバランサー |
| APサーバ（Gunicorn＋Django） | FastAPIコンテナ |
| DBサーバ（PostgreSQL） | PostgreSQLコンテナ |

```
ユーザー → ロードバランサー(GCP)
              ├→ Reactコンテナ（画面）
              └→ FastAPIコンテナ（API）→ PostgreSQLコンテナ（DB）
```

やることは同じだけど、**自分で管理する部分が減り、クラウドに任せる部分が増える**。

---

## 8. 環境隔離 — venvからコンテナへ

**人間**: 開発環境はvenvでいいの？

**AI**: Docker化する前はvenvを使いますが、Docker化した後はvenv不要です。

```
venvの役割:     Pythonの依存ライブラリを隔離する
コンテナの役割:  OS含めて丸ごと隔離する
```

コンテナはvenvの上位互換。venvは「Pythonだけ隔離」、コンテナは「全部隔離」。

**人間**: Docker化してもソース改修は続くじゃん？毎回ビルドし直すの？

**AI**: 開発中はビルドし直しません。**ボリュームマウント**で、ローカルのソースコードをコンテナ内に直接マッピングします。ファイルを保存したら即反映されます。

```yaml
# docker-compose.yml（開発用）
volumes:
  - ./api:/app    # ローカルのソースをコンテナ内に直接マッピング
```

開発体験は今とほぼ変わりません。

---

## 9. プロジェクト間の干渉 — 実体験からの教訓

**人間**: 実際、前のプロジェクトで同じUbuntu上のPostgreSQLが壊れた。

Djangoプロジェクトを複数進めていたとき、同じサーバー上でNginx・Gunicorn・PostgreSQLを共有していたため、プロジェクト間で設定が干渉し、DBが壊れる問題が実際に起きた。

**AI**: コンテナなら各プロジェクトが完全に独立します。他のプロジェクトのDBを壊すことが**物理的に不可能**です。

> **教訓**: ルールで「リソース分離」を定めて対処するのがモノリスの限界。コンテナは**構造で解決**する。

---

## 10. テスト — V字モデルとCI/CDは別物

**人間**: CI/CDで自動テストできるなら、今までのテスト工程はいらなくなるの？

**AI**: いいえ、まったく別物です。

| 観点 | V字モデルのテスト工程 | CI/CDの自動テスト |
|------|---------------------|------------------|
| いつ | システムを作る段階 | 作った後、リリースのたび |
| 目的 | 品質を検証し、バグを見つける | 既に合格済みのテストを再実行し、壊れてないか確認する |
| 期待値の根拠 | 設計書 | 過去に合格したテストケース |

**人間**: 「テスト自動化」って言葉が紛らわしいんだよな。

**AI**: CI/CDがやるのは**回帰テスト（リグレッション）の自動化**です。品質保証そのものの代わりにはなりません。テストコードの品質が低ければ、壊れているのに「Pass」が出て、そのまま本番にデプロイされてしまいます。

> **結論**: V字モデルのテスト工程の精度が、CI/CDパイプラインの信頼性を決める。

---

## 11. CI/CDの選択 — GitHub ActionsとCloud Buildの役割分担

**人間**: CI/CDはだいたい何を使って組むかデファクトスタンダードみたいなものあるよね？

**AI**: GCPでやるなら主な選択肢は4つです。

| ツール | 特徴 | 向いているケース |
|--------|------|----------------|
| **GitHub Actions** | GitHubに組み込み。YAML1つで始められる | 既にGitHub使っている場合 |
| **Cloud Build** | GCPネイティブ。GCPサービスとの連携が楽 | GCPへのデプロイ自動化 |
| **Jenkins** | 老舗。自由度最高だが自分で管理が必要 | 複雑なパイプラインが必要な場合 |
| **GitLab CI** | GitLabに組み込み | GitLab使っている場合 |

**人間**: GitHub使ってるから、GitHub Actionsが自然だね。CIがテストってこと？

**AI**: そうです。

- **CI（継続的インテグレーション）** — pushするたびに自動でテスト・ビルド
- **CD（継続的デリバリー/デプロイ）** — CIが通ったら自動で本番にデプロイ

**人間**: じゃあ、今もうgitとGitHubを使っている状況だから、GitHub Actionsが一番移行しやすいね。

```
git push → GitHub Actions（テスト実行）→ Cloud Build（Dockerビルド・GCPデプロイ）
```

テストはGitHub側、デプロイはGCP側で役割分担。しかもGitHub Actionsは無料枠がある。

---

## 12. 開発フロー — git commitとgit pushの使い分け

**人間**: ローカルでしっかりテストをやりきったものだけをGitHubへプッシュで、あとは自動的にクラウドへ流れるってことか。

```
開発フェーズ（ローカル）:
  コード修正 → git commit（こまめに）
  → V字モデルのテスト工程（単体→結合→システムテスト）
  → 品質確認済み

リリースフェーズ（自動）:
  git push → CI（回帰テスト）→ CD（ビルド・デプロイ）→ 本番反映
```

git commitとgit pushの使い分けが、そのままローカル品質管理とリリースのゲートになる。

---

## 13. インフラ管理 — Terraformで「壊して作り直す」

**人間**: インフラ構築の内容を全部設定ファイルに書き込んでおいて、いつでも同じインフラ環境が構築できるやつね。

**AI**: それがTerraform（IaC: Infrastructure as Code）です。

```
terraform apply   → インフラ構築（VPC、GKE、DNS、LB…全部）
terraform destroy → インフラ全削除（課金停止）
terraform apply   → また同じ環境が復活
```

**人間**: GKEのコストが高くても、学習するときだけapply、終わったらdestroyで解決できるね。

**AI**: 設定ファイルはgit管理されているので何も失われません。「壊して作り直す」が学習手段になります。

**人間**: Terraformってクラウドが変わっても使えるの？

**AI**: はい。Terraformの**書き方・考え方・コマンド**はGCP・AWS・Azure共通です。ただし、**中身の設定はクラウドごとに変わります**。

```hcl
# GCPの場合
resource "google_container_cluster" "main" {
  name     = "my-cluster"
  location = "asia-northeast1"
}

# AWSの場合
resource "aws_eks_cluster" "main" {
  name     = "my-cluster"
  role_arn = aws_iam_role.eks.arn
}
```

やりたいこと（Kubernetesクラスタを作る）は同じでも、設定の書き方が違います。料理に例えると、**レシピのフォーマットは同じだけど食材が違う**ようなものです。

**人間**: 知識はそのまま使えるけど、クラウドが変われば設定も変わると。

> **ポイント**: Terraformを学んでおけば、将来AWSやAzureに移行するときもツールの学び直しは不要。変わるのは設定の中身だけ。

---

## 14. サーバーの箱 — GCE・GKE・Cloud Runの選択

**人間**: Cloud Runじゃだめなの？

**AI**: Cloud Runは便利ですが、Kubernetesが隠蔽されます。Pod構成もログPodもオーケストレーションも全部GCPが裏でやるので、学びになりません。

| サービス | 特徴 |
|---------|------|
| **Cloud Run** | 箱もKubernetesも全部隠蔽（楽だけど学びが少ない） |
| **GKE** | 箱は任せてKubernetesを自分で制御 |
| **GCE** | 箱から全部自分で管理（大変すぎる） |

今回はGKEが正解。

---

## 15. ローカル検証 — minikubeの役割

**人間**: Kubernetesをローカルで動作検証したい場合は？

**AI**: **minikube**を使います。ローカルに1ノードのKubernetesクラスタを作れます。

```
Docker Compose  → ローカルで手軽に開発・テスト
minikube        → Kubernetes設定の動作検証
GKE             → 本番デプロイ
```

同じコンテナイメージを3段階で使い回せます。Pod数の設定、ログ収集Pod、自動復旧、ローリングアップデート——全部ローカルで試してからGKEに持っていけば安全です。

---

## 16. モノリス経験がクラウドネイティブで活きる

**人間**: モノリシックなシステム開発経験があると、新しいアーキテクチャの理解も比較しながら進められるね。

- MVCの3層構造を理解している → マイクロサービスの分割が「何を分けているか」がわかる
- Nginx＋Gunicorn＋Djangoの構成経験 → ロードバランサー＋コンテナの構成が対比で理解できる
- V字モデルのテスト工程 → クラウドでもそのまま使える
- 設計書・ドキュメント管理の型 → Reactの画面遷移管理で活きる

モノリスを知らずにいきなりマイクロサービスから入ると「なぜ分けるのか」がわからない。モノリスの限界を体感した人は「だから分けるのか」と腹落ちする。

## まとめ: モノリス vs クラウドネイティブ 比較表

| 観点 | モノリス（Django） | クラウドネイティブ（React + FastAPI + GKE） |
|------|-------------------|------------------------------------------|
| **アーキテクチャ** | 1つのアプリに全部入り | フロント・バック・DBが独立コンテナ |
| **Webサーバ** | Nginx | Reactコンテナ＋GCPロードバランサー |
| **APサーバ** | Gunicorn＋Django | FastAPIコンテナ |
| **DBサーバ** | PostgreSQL（共有） | PostgreSQLコンテナ（独立） |
| **画面遷移の制御** | サーバー側（views.py） | ブラウザ側（React Router） |
| **遷移の一覧性** | urls.pyで一覧できる | コンポーネントに分散（設計書が重要） |
| **画面の描画** | サーバーがHTML生成 | Reactがブラウザ内で描画（SPA） |
| **データ連携** | テンプレートに直接渡す | JSON API |
| **状態管理** | サーバーセッション | ステートレス＋外部保持（URL・トークン・DB） |
| **環境隔離** | venv（Pythonのみ） | コンテナ（OS含め丸ごと） |
| **プロジェクト分離** | ポート変更＋DB分離（干渉リスクあり） | コンテナ独立（干渉不可能） |
| **開発中のソース反映** | ファイル保存→即反映 | ボリュームマウントで即反映（体験は同じ） |
| **デプロイ** | 手動（Nginx設定、Gunicorn起動…） | git push→CI/CDで全自動 |
| **テスト（品質保証）** | V字モデル | V字モデル（変わらない） |
| **テスト（回帰）** | 手動で再実行 | CI/CDで自動実行 |
| **CI/CD** | なし（手動デプロイ） | GitHub Actions（CI）＋ Cloud Build（CD） |
| **インフラ管理** | 手順書で記録 | Terraform（コードで管理・再現可能） |
| **インフラ再構築** | 手順書を見ながら手動 | `terraform apply` で一発 |
| **スケール** | サーバーごと増やす | 必要なコンテナだけ増やす |
| **可用性** | 手動でサーバー追加 | マルチゾーン設定（アプリ変更なし） |
| **コスト管理** | サーバー常時稼働 | 使わないときは`terraform destroy` |
| **環境再現性** | 手順書頼み | Dockerfile＋docker-compose.ymlで誰でも再現 |
