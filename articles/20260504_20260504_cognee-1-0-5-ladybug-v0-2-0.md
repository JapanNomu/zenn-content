## はじめに

先日、[Claude Code に Cognee グラフ記憶を追加する自作ツールキット](https://zenn.dev/japannomu/articles/20260502_cognee-graph-memory-toolkit) を公開した直後、Cognee 1.0.4 のリリースで **graph DB が KuzuDB から Ladybug DB に置換** されるという大きな変更が入りました。配布物の動作実証バージョンを 1.0.3 のまま放置するわけにもいかないので、Cognee 1.0.5 対応版として **v0.2.0** をリリースしました。

本記事はその v0.2.0（および間に挟まる v0.1.12）で何をやったかを、ソースコード差分・LLM 動作実証マトリクス・速度実測値まで含めて記録します。

リポジトリ:

- 🇬🇧 English: https://github.com/JapanNomu/tools
- 🇯🇵 日本語: https://github.com/JapanNomu/tools-ja

前回告知（v0.1.11）からの主な変更を、バージョンごとに記録します。


## v0.1.12（2026-05-03）— ドキュメント整理

最小限のメンテナンスリリース。動作変更なし。

### Changed

- **GPU 表記の誤解を招く記述を修正**: README の推奨環境表記で、`RTX 4070 12GB` の表現がノート PC 版（VRAM 8GB）を含むと誤読される可能性があったため、デスクトップ版限定であることを明示。
- **CHANGELOG 履歴を整理**: v0.1.0〜v0.1.9 までの内部開発履歴は GitHub リポジトリ初期化時に削除済みだったため、CHANGELOG 上の記述を「クリーンな初回公開を v0.1.10 とする」流れに統一。

このリリースはドキュメント修正だけで、ソースコードへの影響はゼロ。


## v0.2.0（2026-05-04）— Cognee 1.0.5 / Ladybug DB 対応

メインのアップデート。**Cognee 1.0.4 で graph DB が KuzuDB → Ladybug DB に置換されたため、配布物を 1.0.5 対応版として作り直したもの**です。

### きっかけ: Cognee 1.0.4 PR #2755

cognee 公式リポジトリで [PR #2755 `feat(graph): replace kuzu with ladybug`](https://github.com/topoteretes/cognee) がマージされ、Cognee 1.0.4 でデフォルトの graph provider が **`kuzu` → `ladybug`** に切り替わりました。

| 項目 | 内容 |
|---|---|
| デフォルト graph provider | `"kuzu"` → `"ladybug"` |
| 依存パッケージ | `kuzu==0.11.3` → `ladybug==0.16.0` に置換 |
| Python 対応範囲 | `>=3.10,<3.15`（Python 3.14 対応） |
| 後方互換 | 旧 Kuzu provider/import 名はエイリアスとして残存（`KuzuAdapter = LadybugAdapter`）|
| 既存 Kuzu データ | 起動時の自動マイグレーション機能あり（`ladybug_migrate.py` が走る）|
| 設定キー | `shared_kuzu_lock` → `shared_ladybug_lock`（旧名はエイリアス）|

つまり Cognee 側は **後方互換を保ったまま** Ladybug に切り替えた形でした。配布物の本質的な動作は変わらないものの、**「動作実証 Cognee バージョン」表記を 1.0.3 → 1.0.5 に揚げる必要がある** ため、配布物を作り直しました。

### スコープ確定の判断

v0.2.0 では、ソースコード側の影響範囲を機械的に確認した結果、**`src/main_src/` 配下に `kuzu` / `ladybug` の文字列なし**（grep で 0 件）。つまり、当配布物のソースコード自体は変更不要でした。

そのため v0.2.0 のスコープを以下に絞りました:

1. **LLM 動作実証マトリクス再実施**（核心）
2. **ドキュメント修正**（KuzuDB 表記 → Ladybug、Version バンプ、CHANGELOG）

設計〜単体/結合/システムテストの再実施は不要としました。

### Added

- **`knowledge/sample_knowledge/05_graph_memory_operations.md` を新規追加**: Cognee グラフ記憶の運用ノウハウをまとめたサンプルナレッジファイル。具体的には以下の内容:
  - cognify を呼ぶタイミング
  - `remember` vs `save_interaction` の使い分け
  - `search` の type 選択（`CHUNKS` / `GRAPH_COMPLETION` 等）
  - `recall` の auto-routing
  - `forget_memory` / `improve` / `prune` の使い方

  これにより、同梱サンプル数は 4 → 5 ファイルに増加。

### Changed

- **Cognee バージョンを 1.0.3 → 1.0.5 に更新**:
  - 依存: `kuzu==0.11.3` → `ladybug==0.16.0`（自動置換・後方互換エイリアスあり）
  - 動作実証 Cognee バージョン: 1.0.5
  - cognee-mcp バージョン: 0.5.4（変更なし）
- **「KuzuDB」→「Ladybug DB」の表記置換**:
  - `README.md`（機能・技術スタック表）
  - `config/.env.example`（`COGNEE_DATA_PATH` のコメントと storage 説明）
  - `docs/SETUP.md`（動作実証バージョン・pip install 例）
  - `docs/GETTING_STARTED.md`(動作実証 Cognee バージョン・速度実測値)
- **同梱サンプル数の表記を 4 → 5 に更新**:
  - `README.md`（ディレクトリ構造）
  - `docs/GETTING_STARTED.md`（`[N/M]` 形式の投入ログ例も含む）
- **`knowledge/sample_knowledge/03_design_decisions.md` の設計判断ノート整理**: KuzuDB / LanceDB / FastEmbed は Cognee に同梱されて自動的に使用されるものでユーザー選択ではないため、「Cognee を採用した理由」セクション配下に統合。「v0.2.0 = Ladybug DB / v0.1.x = KuzuDB」の注記を追加。
- **`knowledge/sample_knowledge/04_common_errors.md` のエラー対処メモ整理**: ローカル LLM の例示を `llama3.1:8b` から `qwen2.5:14b`（v0.2.0 の動作実証対象）に切替。
- **`harness/rules/cognee_memory_usage.md` の recall 失敗条件記述を更新**: 「`llama3.1:8b` で動かす際に失敗することがある」→「`qwen2.5:14b` 以外のローカル LLM で動かす際に失敗することがある」。

### Fixed（既存ソースコード欠陥 7 件・v0.1.10〜v0.1.12 から残っていたもの）

v0.1.x の既存コードに残っていた欠陥を、ついでに修正しました。**動作影響あり** の修正です。

#### `src/main_src/import_to_graph.py`

- **古い表示行を削除**: `list_targets()` 関数内で、`TARGET_MAP` に定義されていない `comments` ターゲットを表示する行が残っていた（v0.1.10 のリファクタの取り残し）
- **`check_ollama()` を `LLM_PROVIDER=ollama` のときだけ実行**: クラウド API 設定時に不必要に Ollama 接続チェックが走って失敗していた問題を修正
- **`--dry-run` 時は `check_ollama()` をスキップ**: ファイル一覧表示だけのモードで Ollama 接続を要求するのは過剰

#### `src/knowledge_src/import_knowledge.py`

- 上記と同じ `LLM_PROVIDER=ollama` ガードを `check_ollama()` に追加

#### `harness/hooks/auto_remember_user_message.py`

- **矛盾する docstring 行を削除**: 「このサンプルはシンプルな direct-call 版を提供」と書かれていたが、実装は実際には queue アプローチのみだった

#### `harness/hooks/cognee_remember_flusher.py`

- **`remaining` フィルタを `not line.strip()` → `line.strip()` に修正**: 失敗エントリがキューから黙って消えていた内部バグ。データ自体は `failed.jsonl` に保存されていたが、**リトライの起点となるキュー側に正しい中身が残らない**致命的な状態だった
- **不要な `sys.path.insert(main_src)` を削除**: `fastmcp` は配布物 venv の site-packages から直接 import 可能

### Verified（v0.2.0 で実証）

#### LLM 動作実証マトリクス（再実施）

v0.1.x 時点では **MCP ツール 4 種**（`remember` / `search(CHUNKS)` / `search(GRAPH_COMPLETION)` / `recall`）のみが検証対象でした。v0.2.0 では cognee-mcp 0.5.4 のツールを機械的に列挙し、LLM 依存ツールを **8 種** に拡張:

| # | ツール | LLM 依存 | v0.1.x 検証 | v0.2.0 扱い |
|---|---|---|---|---|
| 1 | remember | あり（cognify 走行） | 実施 | 継続 |
| 2 | search(CHUNKS) | なし（ベクトルのみ） | 実施 | 継続（決定論的・参照基準） |
| 3 | search(GRAPH_COMPLETION) | あり | 実施 | 継続 |
| 4 | recall | あり | 実施 | 継続 |
| 5 | **cognify** | あり（エンティティ抽出） | 未実施（マトリクス漏れ） | **追加** |
| 6 | **save_interaction** | あり（structured output） | 未実施（マトリクス漏れ） | **追加** |
| 7 | **improve** | あり（推奨アクション生成） | 未実施（マトリクス漏れ） | **追加** |
| 8 | **forget_memory** | あり | 未実施（マトリクス漏れ） | **追加** |

LLM を使わないツール（`list_data` / `delete` / `delete_dataset` / `prune` / `cognify_status`）はマトリクス対象外。

#### 検証環境とスコープ縮小

- 検証環境: NVIDIA GeForce RTX 4060 Laptop GPU（VRAM 8GB）/ RAM 32GB / qwen2.5:14b（num_ctx=8192）/ Cognee 1.0.5（Ladybug DB）
- 当初は 6 LLM × 8 ツール × 5 回 = 240 回を計画していたが、**qwen2.5:14b で実用性が確認できた時点で他 LLM の検証を対象外に縮小**（時間・GPU 負荷の節約）

#### 結果（qwen2.5:14b・8 ツール × 5 回 = 40 回）

- **35 件成功 / 5 件失敗**
- 失敗 5 件はすべて `save_interaction` ツール起因
- これは **cognee-mcp 0.5.4 と cognee 1.0.5 の API 不整合（後述）** が原因で、当配布物のコード起因ではない

その他のツール（`remember` / `search` / `recall` / `cognify` / `improve` / `forget_memory` 等）は完全動作を実証。

#### Ladybug DB による速度向上

実測値（qwen2.5:14b 環境・参考値）:

| ツール | 実測時間 |
|---|---|
| search(CHUNKS) | 3.2 秒 |
| search(GRAPH_COMPLETION) | 平均 14.6 秒（範囲 12〜18 秒）|
| recall（Q-A・TEMPORAL routing）| 20〜24 秒 |
| recall（Q-B・GRAPH_COMPLETION_COT routing）| 154〜156 秒 |
| improve / forget_memory | 全件即時（数秒以下）|

KuzuDB から Ladybug DB への置換に伴う速度差を実機で確認した値です。比較対象として v0.1.x 時点の実測値は `docs/GETTING_STARTED.md` 末尾の付録に残しています。

### Known Issues（既知の制限事項）

#### `save_interaction` ツールが利用不可

- エラー: `add_rule_associations() got an unexpected keyword argument 'context'`
- 原因: cognee 1.0.5 で `add_rule_associations` の引数が `context` → `ctx` にリネームされたが、cognee-mcp 0.5.4 が未追従（upstream の `topoteretes/cognee` main branch も同状態）
- 代替案: 会話ペアの永続記憶への即時保存は `remember(data="User: ... / Assistant: ...")` で代替可能

`save_interaction` は cognee 側の API リネームに cognee-mcp が追いついていないだけで、**当ツールキット側で対処できる問題ではない**ため、Known Issues として明示しています。

### Migration（v0.1.x からのアップグレード）

- **既存データを保持したい場合**: `pip install -U "cognee[fastembed]==1.0.5"` でアップグレード → 初回 `cognee` 起動時に Cognee 1.0.4 の自動マイグレーション機能で KuzuDB データが Ladybug 形式に変換される（`ladybug_migrate.py` が走る）
- **既存データをリセットしてよい場合**: `forget_memory(everything=True)` で全削除後、Cognee 1.0.5 環境で再 cognify


## まとめ

v0.2.0 は「**Cognee 1.0.4 の Ladybug DB 化に追随しつつ、既存欠陥を整理し、検証マトリクスを 4 ツール → 8 ツールに拡張した版**」です。

設計工程は省略しつつも、機械的に grep / diff してスコープを確定し、LLM マトリクスは縮小して実用性を実証する、という割り切ったやり方で進めました。
