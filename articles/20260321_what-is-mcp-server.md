## 1. 作るものを決める

**人間**: Ubuntu内にMCPサーバとMCPサーバから呼ばれるAPI、APIから接続されるDBがある。それらを全部作ってClaude Codeのチャットから動作確認したい。MCPサーバが提供するのは商品の検索・在庫数を更新・商品の登録の3つ。APIも同じ3機能。DBはテーブル1つ、id/name/category/stockの4項目、データは5レコード。

**AI**: 構成はこうですね。

```
Claude Code のチャット
    ↓（MCP ツール呼び出し）
MCP サーバ（Ubuntu内）
    ↓（HTTP リクエスト）
FastAPI（Ubuntu内）
    ↓（SQL）
SQLite（Ubuntu内）
```

## 2. なぜMCPサーバが必要か

**AI**: 一個質問してもいいですか？MCPサーバが商品検索・更新・登録をするとして、Claude Codeから「`SELECT * FROM products` って SQL 直接書いて」って頼めばSQLiteに繋いでもらえますよね。なんでMCPサーバとAPIをわざわざ間に挟むんでしょう？

**人間**: サプーさんの説明では、Claude CodeはAPIを直接呼べるみたい。それ以外のLLMはできないものもあるみたい。知っている情報としては、LLMがAPIを使う場合の共通規格としてAnthropicのMCPサーバ規格が採用されたらしいということくらい。なぜMCPサーバを経由する必要があるのかは、Claude Codeを使っている身としては理解も腹落ちもできていない状況。

**AI**: その「腹落ちできていない」という感覚が、この記事のいちばん大事な出発点だと思います。

整理するとこういう状況ですよね。

> Claude Code は API を直接呼べる。なのになぜ MCP サーバが必要なのか？

**「Claude Code が API を呼べる」のは、Claude Code が特別だから**です。

Claude Code は Bash も WebFetch も使えるので、`curl http://localhost:8000/products` って直接叩けます。でも、他の LLM（GPT-4o、Gemini など）は同じことができません。

そこで Anthropic が「LLM がツールを使うときの共通規格」として出したのが **MCP**。

- **MCP 以前**: LLM ごとにツールの繋ぎ方が違う。ChatGPT 用に作ったツールは Claude で使えない
- **MCP 以後**: 規格に従って MCP サーバを作れば、どの LLM からでも同じように使える

---

## 3. Claude Code ユーザーにも MCP サーバが必要な理由

**人間**: Claude Code ユーザーに関係ないかどうかはわからない。関係ないのだとしても、いつまでも Claude Code を使うわけではないし、LLM はお客様によって何を使うかわからないから、MCPサーバの共通規格があった方が何かとうれしい。

**AI**: おっしゃっていることは、すごく本質的だと思います。

- 明日には別のエージェントを使うかもしれない
- 作ったツールをお客様に渡すとき、お客様が何のLLMを使っているかわからない
- そのたびに「このLLM用の繋ぎ方」を作り直すのは非効率

だから **MCP サーバを一つ作っておけば、どの LLM からでも繋げる**。一回の投資で済む。

USB の例えで言うと「うちのパソコンは独自端子でも充電できるけど、USB-C で作っておけば誰でも使えるよね」という感覚に近いです。

## 4. curl とは何か。なぜ他のLLMは使えないのか

**人間**: curlコマンドはなんで他のLLMではたたけないのか、curlは何ができるコマンドなの、どうしてClaude Codeはcurlが使えるの？

**AI**:

**curl とは何か**

`curl` は「指定した URL に HTTP リクエストを送るコマンド」です。ターミナルで動くプログラムで、Ubuntu なら最初から入っています。`curl http://localhost:8000/products` と打てば GET リクエストを送れる、ただそれだけのツールです。

**なぜ Claude Code は curl が使えるのか**

Claude Code には「Bash ツール」があります。これは「Ubuntu のターミナルでコマンドを実行する能力」です。`ls`・`python`・`git` など、OS にある任意のコマンドを実際に走らせることができます。

Claude Code は「チャットできる AI」ではなく「**ターミナルに繋がったエージェント**」として設計されているんです。

**なぜ他の LLM は curl を叩けないのか**

ほとんどの LLM は「テキストを受け取ってテキストを返す」だけの存在です。`curl http://...` という文字列を**生成することはできます**。でも実際に**実行することはできません**。コマンドを走らせるには OS の上で動くプロセスが必要で、テキスト処理しかできない LLM にはその仕組みがありません。

| | LLM（一般） | Claude Code |
|---|---|---|
| テキスト生成 | ✅ | ✅ |
| コマンド実行 | ❌ | ✅（Bash ツール） |
| curl で API 呼び出し | ❌（生成はできる） | ✅（実際に実行） |

だから MCP サーバが「**LLM の代わりにコマンドを実行してくれる窓口**」として機能するわけです。

## 5. Claude AI と Claude Code は別物

**人間**: Claude AIも同じようにMCPサーバがなくてもAPIコールできる仕組みなのかどうか。

**AI**:

| | Bashで直接実行 | MCPクライアント |
|---|---|---|
| **Claude Code** | ✅ | ✅ |
| **Claude Desktop** | ❌ | ✅ |
| **Claude.ai web** | ❌ | 拡張中 |

「直接 API コールできる」のは Claude Code だけで、Claude AI は他の LLM と同じ立場です。

> **Claude Code は「エージェント」、Claude AI は「チャット」**

## 7. MCPが解決したこと

**人間**: APIを呼ぶためにはLLMごとに、APIコールの仕組み（ツールと呼んでいるもの）を作らなければならないんだね。それがバラバラだと。

**AI**: そうです、まさにそれです。

**MCP 以前：**
```
ChatGPT  → OpenAI形式のツール定義  → 商品検索API
Claude   → Anthropic形式のツール定義 → 商品検索API
Gemini   → Google形式のツール定義  → 商品検索API
```
同じ API なのに、LLM の数だけツール定義を作らないといけない。

**MCP 以後：**
```
ChatGPT  ┐
Claude   ├→ MCPサーバ（1つ） → 商品検索API
Gemini   ┘
```

MCP サーバを1つ作れば、どの LLM からでも同じように呼べる。

**人間**: そうすると、Claude Code以外のLLMはMCPサーバ共通規格に準拠することによって、実質的にAPIコールできる仕組みをたやすく手に入れられたってわけか。

**AI**: そうです、完璧な理解です。

> **「規格に乗っかるだけで、実行能力を持てる」**

これが MCP の最大のメリットです。

## 8. MCPサーバの役割は3つ。うち1つが普遍的

**人間**: LLMに使われたら困るAPIもあるから、ホワイトリストとしてMCPサーバは機能するわけか。

**AI**: そうです、そこが一番クリーンな答えだと思います。

**MCP サーバの役割は3つ：**

1. **共通窓口**：どの LLM からでも同じ方法で呼べる
2. **代行実行**：API コールできない LLM の代わりに実行する
3. **ホワイトリスト**：LLM に使わせるツールを明示的に制限する

**人間**: 1,2は今だけだと思うんだよ。3は普遍的。

**AI**: 鋭い。1と2はLLMの進化とともに意味が薄れていく。でも3はLLMがどれだけ賢くなっても変わらない。

「強いからこそ制限が必要」という話で、むしろLLMが何でもできるようになればなるほど、ホワイトリストの重要性は上がります。

## 9. MCPサーバの内部構造

**人間**: MCPサーバとはどんな内部構造をしてるのか知りたい。LLMとの窓口のところが気になる。

**AI**: MCP の通信は **JSON-RPC** という形式を使います。

起動時に Claude Code が MCP サーバに「どんなツールを持ってる？」と問い合わせます。

```json
{ "method": "tools/list" }
```

MCP サーバはこう返します。

```json
{
  "tools": [
    {
      "name": "search_products",
      "description": "商品を検索する",
      "inputSchema": {
        "type": "object",
        "properties": {
          "query": { "type": "string" }
        }
      }
    }
  ]
}
```

ユーザーが「りんごを探して」と言ったとき、Claude Code が MCP サーバに送るのはこれです。

```json
{
  "method": "tools/call",
  "params": {
    "name": "search_products",
    "arguments": { "query": "りんご" }
  }
}
```

**この `tools/list` と `tools/call` の形式が共通規格の正体です。**

メッセージの運び方（トランスポート）は主に2種類あります。

| 方式 | 仕組み |
|---|---|
| **stdio**（標準入出力） | MCP サーバをサブプロセスとして起動。stdin/stdout でやり取り |
| **HTTP** | すでに起動しているサーバに HTTP で接続 |

**人間**: つまり、テキスト情報を作ってMCPサーバに投げるだけの仕組みに各LLMが対応すれば済むってことかな。

**AI**: そうです。LLM は「テキストを生成する」のが本業ですから、JSON というテキスト形式の規格との相性は抜群です。

あえて一つ補足すると、LLM 自身が JSON を生成して投げるというより、LLM が「search_products を呼びたい」という意図を持ち、**周りのフレームワーク（Claude Code など）が JSON-RPC メッセージに変換して投げる**という分担になっています。

```
人間 → LLM（意図を判断） → Claude Code（JSON-RPCに変換） → MCPサーバ
```

## 10. MCPクライアントとは何か

**人間**: 今の説明だと必ずエージェントが必要になる気がするけど。

**AI**: 正確に言うと、必要なのはエージェントではなく **MCP クライアント**です。

```
ユーザー → MCPクライアント → MCPサーバ
```

MCPクライアントの中身が何かによって2パターンあります。

```
【LLMを含む場合】
ユーザー →（自然言語）→ [LLM + MCPクライアント] → MCPサーバ
                         Claude Code、Cursorなど

【LLMを含まない場合】
プログラム →（コードで直接）→ [MCPクライアント] → MCPサーバ
                               自作スクリプトなど
```

**人間**: なるほど、Claude Codeは開発するシステムの中に組み込んでMCPクライアントとして動作させることができるのね。そういうエージェントがいなければプログラムでクライアントを自作しなきゃならないと。

**人間**: あ、LLMがMCPクライアント機能を内蔵してるのか、共通規格に準拠するってそういう意味？

**AI**: そうです、まさにそれです。「MCP共通規格に準拠する」とは、その LLM 製品が MCP クライアント機能を内蔵しているということです。

ただ一つだけ細かい補足を：

- **LLM モデル本体**（ニューラルネットワーク）→ テキストを生成するだけ。MCP は関係ない
- **LLM を包むアプリケーション**（Claude Code、Cursor など）→ ここに MCP クライアントが内蔵されている

**人間**: なので、今現在はMCPが共通規格になったんだからLLMがMCPクライアントを内蔵していることを前提として考えていくべきだよね。

**AI**: 2026年現在、MCP が共通規格になった前提で考えると、どの MCP クライアントを使うかは好みや都合の問題であって、技術的な制約ではありません。

今回のプロジェクトで Claude Code をクライアントに使うのは「たまたま手元にあって便利だから」という理由で十分です。

```
Claude Code    ┐
ChatGPT        ├→ 今回作るMCPサーバ → FastAPI → SQLite
Cursor         │
将来のLLM      ┘
```

これが最初に話した「一回の投資で済む」の本当の意味です。

---

## 11. 全体像

**人間**: LLM側にMCPクライアント実装、API提供側にMCPサーバ実装して、間は同じ規約でやりとりするんだな。イメージついてきた。

**AI**: そうです、それが全体像です。

```
LLM製品          間の規約（MCP）        API提供側
─────────────    ────────────────    ─────────────
MCPクライアント  →  tools/list        →  MCPサーバ
                 ←  ツール一覧        ←
                 →  tools/call       →
                 ←  実行結果          ←
                                          ↓
                                        FastAPI
                                          ↓
                                        SQLite
```

「間の規約が共通」だから、左側が Claude Code でも ChatGPT でも、右側が今回作る商品管理サーバでも別の何かでも、繋げられる。
