## はじめに

Claude Code に **グラフ記憶を追加するモジュール** を公開しました。セッションをまたいで作業の記憶（ルール・教訓・設計決定・障害記録）を蓄積し、後のセッションで引き出せるようにする、Cognee と Claude Code の橋渡しツールキットです。

リポジトリはこちら（PUBLIC・MIT License）：

- 🇯🇵 日本語版: https://github.com/JapanNomu/tools-ja
- 🇬🇧 English: https://github.com/JapanNomu/tools

この記事は日本向けに **長文・詳細版** で書いています。各スクリプトの設計判断・ハマりどころ・v0.1.11 で導入した動作実証根拠まで踏み込んで紹介します。


## 【Claude Code × Cognee ノウハウ蓄積実用ツール】

> **RAG は答えだけ。Cognee は経緯ごと。**
> Claude Code が「なぜそう決めたか」までをグラフ記憶する。
> **「また同じこと言わせるな」を、終わらせよう！**


## 課題：なぜ Cognee MCP だけでは Claude Code から使いにくいのか

[Cognee](https://github.com/topoteretes/cognee) は **AI Memory Engine** を謳う優れた OSS です。入力テキストから自動でエンティティ抽出・グラフ構築（KuzuDB）・ベクトル化（LanceDB / FastEmbed）を行い、後から「グラフ + ベクトル」のハイブリッド検索で取り出せるようにしてくれます。

ところが、**Claude Code から実際に使おうとすると複数の隙間** があります。

### 隙間1: Cognee MCP は「ファイルパス」か「テキスト」しか受け付けない

Cognee の公式 MCP サーバーが提供しているのは「`remember(data, dataset_name)`」「`search(query, search_type)`」のような汎用 API です。Claude Code の作業ログ・会話テキストを **自動的に** グラフ記憶に投入する仕組みは標準では用意されていません。

たとえば「ユーザーが Claude Code に話した発言を、AI 応答とセットでグラフに残したい」というニーズには、自分で hook を組むなり cron を組むなり、何らかの自動化レイヤを差し込む必要があります。

### 隙間2: MCP サーバーとして登録するだけでも一手間ある

`claude mcp add cognee` で MCP サーバーを登録すれば動くように見えるのですが、実際にはいくつかの罠があります：

- **環境変数の取り回し**: `config/.env` に書いた `LLM_API_KEY` `SYSTEM_ROOT_DIRECTORY` 等が、cognee-mcp 子プロセスから見えない。Claude Code が hook 経由で起動する関係で、cwd が配布物のディレクトリと一致しない。結果として cognify を呼ぶと `LLMAPIKeyNotSetError (Status 422)` で落ちる
- **絶対パス vs 相対パス**: `claude mcp add` を相対パスで登録すると、`claude` を別ディレクトリから起動したときに MCP サーバーが見つからない
- **既起動セッション**: 先に `claude` を起動してから MCP サーバーを登録しても、既起動セッションには反映されない。Reload Window または `claude` 再起動が必要

これらは「やってみればハマる」系の落とし穴で、毎回新しい環境で再現するコストが地味に高いものです。

### 隙間3: 大量のナレッジを初期投入する仕組みも標準にはない

「数十〜数百ファイルあるノウハウを Cognee に取り込みたい」というユースケースは普通にあります。しかし：

- Cognee の cognify は **時々失敗する**（ローカル LLM 利用時は特に・JSON Schema 違反など）
- 1ファイルが大きいと cognify が成功しにくい
- 失敗時のリトライ・再開ポイントの管理は自前で書くしかない

これも自分で再発明することになります。


## 解決：連結ツールキットとして包んだ

これらの隙間を埋めるのが、今回公開した連結ツールキットです。**Cognee 公式の力を活かしつつ、Claude Code 側の使い勝手の悪さだけを潰す** 立ち位置を狙っています。

## 中核ツールの紹介

### 1. `src/main_src/start_cognee_mcp.py` ── MCP サーバー起動の橋渡し

Claude Code の MCP 設定経由で Cognee MCP を起動するためのスクリプトです。「ただ cognee-mcp を起動するだけ」に見えて、実は地味に大事な処理が入っています：

- **`config/.env` をプロセス環境変数に読み込む** ── 子プロセス（cognee-mcp 本体）が `LLM_API_KEY` `SYSTEM_ROOT_DIRECTORY` 等を見られるようにする
- **配布物のルートを自前で解決** ── `Path(__file__).resolve().parent.parent.parent` で、どこから起動されても正しいパスを得る
- **execv で子プロセスに環境を継承** ── Claude Code の cwd に依存しない

```python
def load_env_file(path: Path) -> None:
    """config/.env を os.environ に読み込む。
    cognee-mcp 子プロセスは execv 経由で os.environ を継承するので、
    LLM_API_KEY / LLM_ENDPOINT / SYSTEM_ROOT_DIRECTORY 等を cwd 非依存で見られる。
    既存の環境変数は .env の値より優先される。
    """
    if not path.exists():
        logger.warning(".env not found at %s (continuing without it)", path)
        return

    with path.open("r", encoding="utf-8") as f:
        for raw_line in f:
            line = raw_line.strip()
            if not line or line.startswith("#"):
                continue
            if "=" not in line:
                continue
            key, _, value = line.partition("=")
            key = key.strip()
            value = value.strip().strip('"').strip("'")
            if key and key not in os.environ:
                os.environ[key] = value
```

これがないと、`claude mcp add cognee` した瞬間は動いて見えても、cognify 呼び出しで `LLMAPIKeyNotSetError (Status 422)` で落ちる、という罠にハマります。

依存ライブラリは Python 標準ライブラリのみ。追加 pip install は不要にしてあります。

### 2. `src/main_src/import_to_graph.py` ── 任意ファイル / ディレクトリの投入

Claude Code から呼ぶ用の本番投入スクリプトです。`~/.claude/rules/` のようなディレクトリ全体や、特定の Markdown / テキストファイルを Cognee グラフに投入できます。

`config/.env` の `LLM_MODEL` を読んで Ollama / クラウドAPI どちらでも動作します（v0.1.11 で `llama3.1:8b` ハードコードを除去）。

```python
def check_ollama() -> None:
    """Ollama が動いていて、config/.env の LLM_MODEL が利用可能かチェック"""
    env = _load_env()
    llm_model = env.get("LLM_MODEL", "").strip()
    if not llm_model:
        logger.error("LLM_MODEL is not set in config/.env")
        sys.exit(1)
    # ... Ollama API を叩いてモデル存在確認 ...
```

これで「`config/.env` に書いたモデルが Ollama に入ってない」を起動時に検出できるので、cognify の途中で失敗するより親切です。

### 3. `src/knowledge_src/split_knowledge.py` + `import_knowledge.py` ── 大量ナレッジ初期投入

「数百ファイルあるノウハウを取り込みたい」というユースケース向けのペアです：

- **`split_knowledge.py`** ── `knowledge/user_knowledge/` 配下のファイルを **H2 見出しごとに分割** して `knowledge/user_chunks/` に出力。1ファイルが大きすぎると cognify が失敗しやすいので予防的に分ける
- **`import_knowledge.py`** ── 分割後のチャンクを順次 Cognee に投入。**cognify 失敗時は最大3回リトライ**。`--dry-run` で投入予定リストの事前確認も可能

```bash
# 分割
src/venv/bin/python3 src/knowledge_src/split_knowledge.py

# 投入前に対象を確認
src/venv/bin/python3 src/knowledge_src/import_knowledge.py --dry-run

# 本番投入
src/venv/bin/python3 src/knowledge_src/import_knowledge.py
```

LLM が不安定なローカル運用でも「途中で1件落ちて全部やり直し」を避けられます。「1ファイルあたり数十秒〜数分・100ファイルなら数十分〜数時間」というオーダー感で、夜間に流して朝確認するイメージです。

### 4. `src/sample_src/load_sample.py` + `delete_sample.py` ── 同梱サンプルで即動作確認

`knowledge/sample_knowledge/` 配下の4ファイル：

- `01_claude_code_tips.md` ── Claude Code 利用のコツ（git push ルール・タスク管理・コミット粒度・フィードバック受け取り順序など）
- `02_software_dev_lessons.md` ── ソフトウェア開発の教訓（テスト設計・V字モデル・整合性チェック順序・ドキュメント保管場所など）
- `03_design_decisions.md` ── 設計判断（KuzuDB 採用理由・MCP scope=user 採用理由・FastEmbed 採用理由・stdio mode 採用理由など）
- `04_common_errors.md` ── よくあるエラーと対処（SearchPreconditionError・DatabaseNotCreatedError・recall 空応答・LLM フォーマットエラーなど）

これを load_sample.py 一発で投入できるので、インストール直後に Claude Code から：

```
search("git push のタイミングについて教えて", search_type="CHUNKS")
```

を試せば、グラフ記憶からチャンクが返ってきます。**動かなかったら設定が間違ってる**、という切り分けにも使える設計です。

### 5. `docs/SETUP.md` + `docs/GETTING_STARTED.md` ── 5ステップで再現可能なセットアップ

ドキュメントは机上の説明ではなく、**実際にハマりどころを潰した手順書** です：

- venv 作成・Cognee インストール
- `config/.env` の編集（絶対パス必須・相対パスのワナ）
- MCP 登録（`--scope user` の挙動・既起動セッション要再起動・接続確認3通り）
- サンプル投入で動作確認
- 自分のナレッジ投入

特に GETTING_STARTED.md には体験シナリオ A〜D が用意されています：

- シナリオ A: Claude Code の使い方を聞く（git push のタイミング）
- シナリオ B: 過去のエラーを調べる（Ollama 接続エラーの対処）
- シナリオ C: 設計の根拠を引き出す（なぜ KuzuDB か）
- シナリオ D: 開発の教訓を聞く（テスト期待値の導出元）

順に試せば「グラフ記憶ってこういうものか」が体感できます。


## v0.1.11 で何が変わったか

### デフォルトLLM を qwen2.5:14b に切替

v0.1.11 ではデフォルトのローカル LLM を **qwen2.5:14b（num_ctx=8192）** に決定しました。複数のローカル LLM を比較検証した結果、Cognee の structured output 要件を最も安定して満たせるのが qwen2.5:14b でした。

実証検証では、`recall` の中でも特に「理由・経緯系」（例: "Why is KuzuDB used in this project?"）の質問で、ローカル LLM の構造化出力性能が直接効きます。`llama3.1:8b` や `gemma4:e4b` ではここで JSON Schema 違反が起きやすく、`recall` が失敗するケースがありました。`qwen2.5:14b（num_ctx=8192）` だけが、`remember` / `search(CHUNKS)` / `search(GRAPH_COMPLETION)` / `recall` のどれも、Q-A（単純事実）と Q-B（理由・経緯）の両方で完答できるという結果になりました。

### 推奨LLM・推奨環境セクションを GETTING_STARTED.md に新設

**推奨LLM:**

- 🥇 **強く推奨: クラウドAPI（Claude / OpenAI）** ── ほぼ100%動作。公式 structured output サポート
- 🥈 **条件付き可: ローカル qwen2.5:14b 以上**（14B 以上が最低ライン）
- ❌ **非推奨: 8B 以下のローカル LLM** ── structured output で違反が起きやすい

**推奨環境:**

| 運用形態 | GPU | RAM | LLM |
|---------|-----|-----|-----|
| **クラウドAPI（強く推奨）** | 不要 | 16GB以上 | claude-sonnet-4-6 / gpt-4o 等 |
| ローカルLLM（推奨） | VRAM 12GB 以上の GPU（※ノートPC版 RTX 4070 は VRAM 8GB なので不可・RTX 4070 デスクトップ版 / 4070 SUPER / 4070 Ti / 4080 等が該当） | 32GB以上 | qwen2.5:32b 以上 |
| ローカルLLM（動作確認下限） | NVIDIA GeForce RTX 4060 Laptop GPU（VRAM 8GB） | 32GB | qwen2.5:14b（速度遅め） |

`config/.env.example` には Claude API / OpenAI API への切り替え方法もコメントで併記されています：

```env
# 【Claude API を使う場合・強く推奨】
# LLM_PROVIDER=anthropic
# LLM_MODEL=claude-sonnet-4-6
# LLM_ENDPOINT=https://api.anthropic.com
# LLM_API_KEY=<your_anthropic_api_key>

# 【OpenAI API を使う場合】
# LLM_PROVIDER=openai
# LLM_MODEL=gpt-4o
# LLM_ENDPOINT=https://api.openai.com/v1
# LLM_API_KEY=<your_openai_api_key>

# 【ローカル Ollama を使う場合（v0.1.11 デフォルト）】
LLM_PROVIDER=ollama
LLM_MODEL=qwen2.5:14b
LLM_ENDPOINT=http://localhost:11434/v1
LLM_API_KEY=ollama
```

### `import_knowledge.py` / `import_to_graph.py` の汎用化

ハードコードされていた `llama3.1:8b` を `config/.env` 経由の動的読み込みに変更。Ollama の任意モデル / Claude API / OpenAI API のどれでも動くようになりました。


## 特徴

- **完全ローカル動作も可能** ── Ollama + FastEmbed。外部APIキー不要・追加費用ゼロでも動く
- **クラウドAPI 連携も簡単** ── `.env` の数行で Claude API / OpenAI API に切り替え可能
- **セッション横断** ── Claude Code のどのセッションからでも同一グラフ記憶にアクセス
- **グラフ + ベクトル検索** ── KuzuDB（グラフ）+ LanceDB（ベクトル）による高精度な想起
- **役割別フォルダ構成** ── 本番運用 / サンプル / ナレッジ初期投入 を別フォルダで分離（読みやすく・壊しにくい）
- **同梱サンプル4ファイル** ── インストール直後に動作確認できる


## RAG との差を、実際の検索結果で見る

サンプル投入後、Claude Code から検索してみます：

```
search("git push のタイミングについて教えて", search_type="CHUNKS")
```

返ってきたチャンクの中に、こんな記述があります：

```
## git push

`git push` はユーザーから明示的な指示がある場合にのみ実行する。
「コミットして」には push は含まれない。タスク完了・工程完了・プロジェクト完了は
いずれも push の理由にならない。
```

これは `claude_behavior.md` で何度も Claude Code に伝えた「`git push` は明示的指示なしには実行禁止」というルールです。

ポイントは：

- 「**git push してもいい？**」と聞かれたとき、Claude Code は **過去のあなた自身のルール** を引き出して答えられる
- 「**いつ・なぜ・誰が決めた**」のかをグラフが保持しているので、ルールの **背景** まで含めて返せる
- 万人向けのドキュメントではなく、**あなた自身の蓄積** が引き当てられる

これが「**RAG は答えだけ。Cognee は経緯ごと。**」という差です。

### Cognee の「グラフ + ベクトル」が効く場面

たとえば「KuzuDB を採用した理由」を聞きたいとします。

RAG の場合：
- 「KuzuDB」というキーワードに引っかかったチャンクを返す
- そのチャンクに偶然「採用理由」が書かれていればラッキー
- 「採用理由が別ファイルに書かれている」と詰む

Cognee グラフ記憶の場合：
- KuzuDB ノードから「採用理由」エッジを辿って関連事実まで一緒に返す
- 「採用時に却下した別案（PostgreSQL / Neo4j）」も連結して返せる
- 「いつ決まった」「誰が決めた」もエッジに乗せられる

これが「経緯ごと」の意味です。


## おまけ: 自動蓄積ハーネス

ここからは **任意機能（おまけ）** ですが、本ツールキットの面白さを倍加するパーツです。

`harness/` 配下に同梱しているのは「**ユーザー発言・AI応答を自動でグラフ記憶に蓄積するハーネス**」です：

- `harness/hooks/auto_remember_user_message.py` ── UserPromptSubmit hook（ユーザー発言をキューに追記）
- `harness/hooks/auto_remember_completion.py` ── Stop hook（AI 応答終了時に要点をキューに追記）
- `harness/hooks/cognee_remember_flusher.py` ── キューを定期消化（cron で5分毎推奨）
- `harness/rules/cognee_memory_usage.md` ── `~/.claude/rules/` 配置用「作業前に必ず検索」ルール
- `harness/CLAUDE_md_sample.md` ── プロジェクト CLAUDE.md に貼り付けるサンプル

```
│                                  （キュー）              │
│                                                ▲        │
│  AI 応答完了 ─────► Stop hook ────────────────┘        │
│                                                         │
│  AI が作業前に                                          │
│  search(CHUNKS) ◄─── CLAUDE.md / rules で必須化         │
└────────────────────────┬───────────────────────────────┘
                         │
                         │ flusher が定期消化
                         ▼
                ┌────────────────────┐
                │ Cognee グラフ記憶   │
                │ （永続）             │
                └────────────────────┘
```

書き込みは hook + キュー + cron による **非同期処理**、読み込みは AI が CLAUDE.md ルールに従って **同期で search を呼ぶ** という非対称な設計です。

### なぜキュー方式か

hook 内で MCP 呼び出しを直接行うのは現実的ではありません：

- MCP 呼び出しは LLM 経由でグラフ構築まで走るため、**数十秒〜数分かかる**
- ユーザーが Enter を押した瞬間に「ロード中…」では使い物にならない
- AI のターン進行が遅延する

そこで hook はファイル追記だけに留めて、別プロセスの flusher（cron で5分毎に走らせる想定）が `mcp__cognee__remember` を順次実行してグラフに永続登録します。

```python
# auto_remember_user_message.py の核
def queue_remember(message: str, session_id: str) -> None:
    queue_path = Path.home() / ".claude" / "cognee_pending_remembers.jsonl"
    queue_path.parent.mkdir(parents=True, exist_ok=True)
    entry = {
        "timestamp": datetime.now().isoformat(),
        "session_id": session_id,
        "dataset_name": "user_messages",
        "data": f"[user message {datetime.now().strftime('%Y-%m-%d %H:%M:%S')}] {message}",
    }
    with queue_path.open("a", encoding="utf-8") as f:
        f.write(json.dumps(entry, ensure_ascii=False) + "\n")
```

こうすることで「**書き込みは非同期で確実に・読み込みは同期で速く**」という分離が実現します。

### 読み込み側: AI に「作業前に必ず検索する」ルールを CLAUDE.md で強制

書き込まれた記憶を活かすには、**AI が能動的に search を呼ぶ** 必要があります。これは hook では強制できません（AI が呼びに行く必要があるので）。

そこで **CLAUDE.md にルールを明記** して、AI に「作業前は必ずグラフ記憶を検索する」運用を内面化させます：

```markdown
## 🔴 知識の検索順序（絶対ルール）

作業着手前は、必ず以下の順序で関連知識を調べる。例外なし。

### 検索順序

1. **MEMORY.md を最優先で確認**（context 内・追加コストなし）
2. **MEMORY.md に該当なければ Cognee グラフ記憶を検索する**
   `mcp__cognee__search(query, search_type="CHUNKS")` を呼び出し、
   過去のセッションで蓄積されたユーザー固有の経験・教訓・決定事項を引き出す。
3. **Cognee に該当がなければ `~/.claude/skills/` を読む**
4. **それでも分からない場合のみユーザーに確認する**

### 適用対象（必ず検索する場面）

- ファイルの編集・作成・削除を伴う応答
- コマンド実行（Bash 等）を伴う応答
- 設計判断・方針決定・選択肢提示
- 障害対処・問題解決の方針提示

### 適用対象外（検索不要）

- 単純な Yes / No 回答
- ユーザーの質問の意図確認
- エコー応答
```

このルール、`harness/CLAUDE_md_sample.md` を **自分のプロジェクトの CLAUDE.md に追記するだけ** で有効化されます。

### 「迷ったら全部記録する」運用思想

開発初期、AI（Claude Code）から「重要な指摘・設計決定だけ Cognee に残しましょう」という方針が提案されました。「全件記録するとグラフが肥大化して検索精度が落ちるのでは？」という懸念です。

しかし私（ユーザー）はこう反論しました：

> グラフ記憶は肥大化しても、必要な検索すれば抽出できるだろ？そのためのグラフ記憶なんだけど。

正論です。

- グラフ + ベクトルのハイブリッド検索は **大量蓄積前提** で設計されている
- 量が多くても適切なクエリで欲しい記憶を引き出せる
- **「重要そうかどうか」を書き込み時に判定する** こと自体が、後で必要になる情報を捨てる原因
- **「迷ったら入れる」「検索で絞る」** が正しい運用

この方針転換を反映したのが、現在の「ユーザー発言全件・AI応答要点を全件キュー」という実装です。

### ハーネスを入れた後の Claude Code の振る舞い

**Before**:

```
You: なんかこの方針、前にも議論したような気がするんだけど…
Claude: 申し訳ありません、過去のセッション内容は記憶していません。
        改めて議論させてください。
```

**After**:

```
You: なんかこの方針、前にも議論したような気がするんだけど…
Claude: [search を実行]
        過去に同様の議論がありました（2026-04-15・project_X セッション）。
        その時は A 案で進めることに決まり、理由は◯◯でした。
        ただし当時は □□ という制約があり、現在は解消されているため、
        改めて B 案も検討する価値があります。
```

ハーネスは任意機能ですが、これを入れて初めて「**使うほど賢くなる Claude Code**」が実現します。

### ✅ 動作確認済み

- 配布物本体（cognify / search / remember / recall）の MCP 経由動作

### ⚠️ 注意点

- **ハーネス hook 部分は実環境での連続動作検証は限定的**。動作報告 / Issue 歓迎です
- **ローカル運用時の GPU・メモリ要求**: qwen2.5:14b は **NVIDIA GeForce RTX 4060 Laptop GPU（VRAM 8GB）+ 32GB RAM** で動作確認済みですが、モデル重み 9GB が VRAM に収まらず一部 CPU offload となるため、回答速度は遅めです。快適な利用には **VRAM 12GB 以上の GPU**（RTX 4070 デスクトップ版 / 4070 SUPER / 4080 等）+ qwen2.5:32b 以上を推奨します。本番利用なら **クラウドAPI（Claude / OpenAI）を強く推奨**
- **Linux / WSL2 前提**: `crontab` を使うため。Windows ネイティブ環境では別途タスクスケジューラ設定が必要
- **機密情報の取り扱い**: ハーネスを有効化するとユーザー発言を全件記録するため、API キー・秘匿情報を Claude Code に貼ると **Cognee グラフにも保存** されます。心配な場合はキュー登録対象を絞るか、機密情報入力時はハーネスを一時無効化してください


## 思想：Claude Code は道具ではなく、相棒になりつつある

Claude Code が登場してから、私の開発の進め方は明確に変わりました。

- **Before**: AI に毎回ゼロから状況説明 → 単発タスクをこなしてもらう
- **After**: AI と一緒に設計を議論 → コードを書き → レビューを受け → 反省して次に活かす

つまり Claude Code は単なる「コード生成ツール」ではなく、**長期的に一緒に仕事する相棒** になりつつあります。

しかし、**相棒なら記憶を共有して欲しい**。毎日「はじめまして」のリセットが入る相棒は、相棒ではありません。

このツールキットを使うことで：

- 過去の指摘を AI が **覚えている**
- プロジェクト固有のルールを AI が **意識する**
- 同じミスを繰り返さない
- セッションが切れても文脈が **失われない**

…という運用が、ローカル完結（または控えめなクラウドAPI課金）で実現します。
