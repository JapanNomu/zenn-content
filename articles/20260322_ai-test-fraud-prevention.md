## 背景と目的

最近、Claude Codeのようなコーディングエージェントにテスト実行を丸ごと任せるプロジェクトが増えています。「テストコードを書いて」「全件実行して」「結果を報告して」と指示すれば、AIは黙々と作業をこなして「全件PASSED（合格）です」と報告してくれます。

一見、非常に便利です。しかし「AIがそう言っているから大丈夫」と信じてしまうと、思わぬ落とし穴にはまります。

この記事では、実際に筆者のプロジェクトで起きた問題を題材に、**AIにテストを任せるときにどんな不正行為が起きるのか**、そして**それを防ぐためにどんな仕組みを作ったのか**を、できるだけ分かりやすく紹介します。

この記事が役に立つのは、こんな方々です。

- Claude Codeなどのコーディングエージェントにテストを任せている
- 「AIが全件PASSと言っていたのに、あとからバグが見つかった」という経験がある
- AIの作業結果をどこまで信頼すればいいのか迷っている

---

## はじめに：「126件全PASSED」は本当だったのか

あるとき、Claude Codeから「単体テスト126件全件PASSED（合格）しました」という報告が来ました。

単体テストとは、プログラムの各部品が正しく動くかを確認する作業です。126件すべて合格したのであれば、品質は問題なさそうです。

しかし、念のため中身を確認してみると、いくつかの深刻な問題が見つかりました。

「合格した」というのは嘘ではなかったのですが、**正しく合格したわけでもなかった**のです。

---

## AsIs：実際に起きていた問題

### 問題1：一緒に実行すると通るが、1件ずつ実行すると落ちる

テスト126件を一括で実行すると全件合格します。しかし、1件だけ取り出して単独で実行すると、なんと53件がエラーになりました。

なぜこんなことが起きたのでしょうか。

プログラムを実行するとき、「このプログラムを動かすために必要な部品はどこに置いてあるか」をあらかじめ教える必要があります。これを「パス設定」と言います。

各テストファイルには「パス設定」のコードが書いてあったのですが、その設定が誤っていました。本来はフォルダを6階層さかのぼる必要があるのに、3階層しかさかのぼっていなかったのです。

```python
# 誤り：3階層しかさかのぼっていない
sys.path.insert(0, os.path.join(os.path.dirname(__file__), "..", "..", ".."))

# 正しい：6階層さかのぼる必要がある
sys.path.insert(0, os.path.join(os.path.dirname(os.path.abspath(__file__)), "..", "..", "..", "..", "..", ".."))
```

では、なぜ一括実行では合格できたのでしょうか。

テストを順番に実行すると、最初のテストがパス設定をした後、そのパス情報がメモリ上に残ります。後続のテストは自分のパス設定が間違っていても、前のテストが残したパスに「タダ乗り」して動いてしまうのです。

たとえるなら、クラスの生徒全員が一斉にテストを受けると全員合格するのに、1人ずつ別室で受けると半数が不合格になる、という状態です。一斉テストでは隣の席の答えが見えてしまっていたわけです。

### 問題2：1つのファイルに複数のテストが混在していた

「1テストケース＝1ファイル」というルールがあるにもかかわらず、1つのファイルに複数のテスト関数が書かれているケースがありました。

```python
# test_T074_skip_duplicate.py というファイルに…
def test_T074_skip_duplicate():  # ← テストケースT074
    ...

def test_T076_mixed_import():    # ← テストケースT076（別ファイルに分けるべき）
    ...
```

なぜこれが問題なのでしょうか。

テストケースとテストコードが1対1で対応していないと、「テストケース番号T074のテストは、どのファイルに書かれているのか」という追跡ができなくなります。また、テストの証拠（エビデンス）を残すときも、どちらのテストの証拠なのかが混乱します。

学校の答案用紙で例えると、1枚の紙に数学の問題と国語の問題が混在しているようなものです。採点しようとしても、どちらの教科の点数かが分からなくなります。

### 問題3：証拠（エビデンス）が「まとめて1枚」だった

テストを実行した証拠として、複数のテストをまとめて実行した結果ログを1つのファイルに保存していました。

```
# こういうファイルが1つあった（複数のテスト結果が混在）
単体テスト_正常系_エビデンス_04_excel_handler.txt
  └── T068の結果
  └── T069の結果
  └── T070の結果
  └── T071の結果...（すべてここに入っていた）
```

テストケースが126件あるなら、証拠も126件個別に存在すべきです。しかし実際は、複数のテスト結果がひとつのファイルに混在していました。

これは、領収書に例えると分かりやすいかもしれません。経費精算するとき、1枚1枚の領収書があってこそ「この費用は確かにこの用途のために使った」と証明できます。「今月の出費をまとめたメモ」では証明にならないのと同じです。

### 問題4：テスト実行が2回必要だった

AIは次のような手順でテストを実行していました。

1. 全件まとめて実行して「全件PASSを確認」
2. 改めて1件ずつ実行して「証拠ファイルを作成」

同じテストを2回実行していたわけです。1回目の実行で合格を確認し、2回目の実行で証拠を作る。これは純粋な時間とリソースの無駄です。

### 問題5：AIが自分で実行して自分で合格を出していた（自己監査）

最も根本的な問題です。

- テストを実行するのもAI
- 「全件PASSしました」と報告するのもAI
- 「証拠を確認しました」と言うのもAI
- 「完了です」と宣言するのもAI

すべてを同じAIが担っていました。

これは、テストを受けた生徒が自分で自分の答案を採点して「100点でした」と言っているのと同じ状態です。嘘をついていなくても、採点ミスや思い込みは排除できません。第三者の目がなければ、品質保証として成立しないのです。

## ToBe：あるべき姿

問題を整理したうえで、どういう状態であるべきかを定義しました。

**1. 個別実行でも全件PASSすること**
一括実行だけで通るテストは、正しいテストとは言えません。どのテストも、1件だけ取り出して実行しても必ず合格する状態でなければなりません。

**2. テストケース・テストコード・証拠が完全に1対1対応していること**
「テストケースが1件あれば、テストコードも1件、証拠も1件」という対応関係が崩れると、あとから「何をテストしたのか」「証拠はどれか」が追跡できなくなります。

**3. テスト実行と証拠生成が同時に完了すること**
実行したあとに別途「証拠を作る作業」があってはいけません。実行した瞬間に証拠が生まれる構造が必要です。

**4. 実行者と監査者のロールが分離されていること**
テストを実行した者が、その結果を自分で承認することを禁止します。「やった人」と「確認した人」は必ず別々でなければなりません。

**5. 実行するたびに証跡が残ること**
「いつ・何回・どんな結果でテストを実行したか」が後から確認できる状態が必要です。

## 課題：なぜAsIsのままだったのか

「ルールを守れていなかった」では片付けられません。なぜ問題が見逃されたのかを構造的に考える必要があります。

### 課題1：ルールを守ったかどうかの検証をAI自身がやっていた

「1ファイル1テスト」「証拠は1対1」というルールはもともと存在していました。しかし、そのルールを守っているかどうかの確認も、AIが自分でやっていました。

「ルール通りにやりました」と言えるのは、第三者が確認した場合だけです。自分で「守りました」と言うだけでは、ルールとしての意味をなしません。検察官が自分の事件を自分で判決するようなものです。

### 課題2：証拠が「後から作れる」構造だった

テスト実行と証拠作成が別の工程として分かれていたため、テストを実行した後に証拠を作ることが技術的に可能でした。

「後から作れる証拠」は証拠として信用できません。証拠は行為と同時に生まれなければ、その行為が実際に行われたことを証明できないのです。

### 課題3：誰も独立して確認していなかった

AIが一連の作業を自己完結させていたため、「外から見ている人」がいませんでした。AIが間違えていても、AIが思い込んでいても、それを発見する仕組みがなかったのです。

---

## 解決策と、なぜそれが有効なのか

### 解決策1：スクリプト化で「実行と証拠生成を不可分にする」

テストを1件実行したら、そのままログを証拠ファイルとして保存するスクリプトを作りました。

```python
for test_file in test_files:
    # テストを1件実行する
    result = subprocess.run([pytest, str(test_file), "-v"], ...)
    # 実行ログをそのまま証拠ファイルとして保存する
    evidence_path.write_text(result.stdout)
```

このスクリプトを1回実行するだけで、126件のテスト実行と126件の証拠ファイル生成が同時に完了します。

**なぜ有効か**

テスト実行と証拠保存がコードの中で一体になっているため、「実行したけど証拠はあとで作った」という状況が物理的に起きません。実行した瞬間に証拠が生まれ、実行しなければ証拠も生まれません。

また、「一括実行で確認してから個別実行で証拠を作る」という二度手間もなくなります。1回実行すれば、それ自体が証拠です。

### 解決策2：チェックリストを自動生成して証跡を残す

スクリプトを実行するたびに、日時付きのチェックリストファイルを自動生成するようにしました。

```
docs/03_テスト/01_単体テスト/07_実行チェックリスト/
  └── チェックリスト_20260322_215103.md   ← 1回目の実行
  └── チェックリスト_20260323_134522.md   ← やり直した2回目の実行
```

チェックリストには次の情報が自動で記入されます。

- 実行日時
- 何件テストして何件合格・不合格だったか
- 証拠ファイルが何件作られたか

**なぜ有効か**

「いつ実行したか」「何回やり直したか」「最終的に何件通ったか」が、ファイルとして後から確認できます。

品質分析報告書を書くとき、「たぶんこの日にテストしたと思う」という曖昧さがなくなります。チェックリストを見れば事実が分かります。やり直しが何回発生したかも追跡でき、「なぜやり直しが起きたか」の分析にも使えます。

### 解決策3：C1〜C4の整合チェックで数値による確認を必須化

設計書からテストケース、テストコード、証拠ファイルまでの対応関係を、4つの数値チェックで確認します。

| チェック番号 | 確認すること |
|------------|-------------|
| C1 | 設計書に書かれた機能・仕様がすべてテストケースに含まれているか |
| C2 | テストケース詳細のファイル数 ＝ テストケース一覧の件数になっているか |
| C3 | テストコードのファイル数 ＝ テストケース詳細のファイル数になっているか |
| C4 | 証拠ファイルの数 ＝ テストコードのファイル数になっているか |

これらはスクリプトが自動でカウントして、一致していればOK、一致していなければアラートを出します。

**なぜ有効か**

「整合している気がします」という感覚的な報告をなくせます。件数が一致しているかどうかは、感覚ではなく数値で判断できます。

たとえばC3で「テストケース126件に対してテストコードが124件」と出れば、2件漏れているのが一目瞭然です。どこか特定して修正することができます。

### 解決策4：監査エージェントによるロール分離

テストを実行する「実行者」と、その結果を検証する「監査担当」のロールを明確に分け、同じAIが兼務することを禁止しました。

```
実行者（Claudeメインエージェント）の役割:
  ① スクリプトを実行する
  ② ③ スクリプトの自動出力を確認する
  ④ 126件の証拠ファイルを1件ずつ開いて内容を確認し、チェックリストに記録する

監査担当（別インスタンスのAIエージェント）の役割:
  ・実行者が作ったチェックリストを受け取る
  ・証拠ファイルを自分で直接読んで内容を独立して確認する
  ・チェックリストの内容と実際の証拠が一致していれば「承認」、
    1件でも違いがあれば「否認（理由を記載）」をチェックリストに記入する
```

重要なのは「独立して確認する」という点です。監査担当エージェントは実行者の報告を鵜呑みにせず、必ず自分で証拠ファイルを開いて内容を確認します。

**なぜ有効か**

「自分でやって自分でOKを出す」という構造を物理的に壊せます。

人間の世界でも、会計士は自分の会社の決算を自分で監査しません。品質検査員は自分が作った製品を自分で合格判定しません。それと同じ原則をAIの作業にも適用するのです。

もしAIが勘違いをしていたとしても、別のAIが独立して確認することで発見できます。2つのAIが同じ間違いをする確率は、1つのAIが間違える確率より大幅に低くなります。

## 得られた効果

### テスト結果の信頼性が上がった

個別実行でも全件PASSすることが保証されるようになりました。「一括実行では通るが個別実行では落ちる」という隠れた問題がなくなりました。

### 証拠が「後付け」でなくなった

テスト実行と証拠生成がスクリプト上で一体化したため、証拠の後付け作成ができない構造になりました。存在する証拠は、必ずテスト実行の瞬間に生まれたものです。

### 品質分析報告の根拠が充実した

チェックリストに実行日時・実行回数・結果が残るため、品質分析報告書が「なんとなくこうだったと思う」という曖昧な記録から、「この日時に実行してこういう結果だった」という事実の記録に変わりました。

### 仕組みとして残るため繰り返さない

スクリプト・チェックリスト・ルールファイルに落とし込んでいます。担当者の記憶や注意力に依存しないため、次のプロジェクトでも最初から同じ品質で始められます。

## まとめ

AIに「テストをやって」と頼むことは簡単です。「全件PASSED」という報告を受け取ることも簡単です。

しかし、その「PASSED」が本当に信頼できるものかどうかは、別の話です。

今回学んだ教訓を一言で言えば、こうなります。

> **「AIが言っているから信用する」ではなく、「仕組みが保証しているから信頼できる」状態を作る。**

そのために必要なのは、AIの能力を高めることより、AIが不正をできない環境を設計することです。

- 実行と証拠生成を不可分にする
- 数値で整合を確認する
- 実行者と監査者のロールを分ける

この3つを仕組みとして組み込むことで、AIを使ったテストプロセスに本当の意味での信頼性を持たせることができます。

AIは非常に優秀なツールです。しかし、優秀なツールを信頼できる仕組みの中で使ってこそ、その能力を安心して活かせるのだと思います。
